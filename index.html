Why Docker?
• It eliminates the age old problem of "Trust me bro, it runs !!!"
• Docker standardizes the runtime environment
• Eliminates dependency conflicts
• Ensures consistency across systems
What is Docker?
• Docker enables containerization, packaging applications with all dependencies into a single unit
• Containers include everything required to run, ensuring consistency across environments
• Docker provides a lightweight alternative to virtual machines for development and deployment
VMs vs Containers
    Virtual Machines
• Full OS per instance
• Heavy and slower boot time
• Hardware-level isolation
   Containers
• Share host OS kernel
• Lightweight and fast
• Process-level isolation
# Containers = Faster, lighter, more scalable$
What is a Container?
• A container is a Linux process with an isolated view of the system
• Uses namespaces for isolation and cgroups for resource control
• File System Layers → efficient storage using layered filesystem
• Shares the host kernel → making it lightweight and efficient

Linux Primitives Behind Containers
• Docker uses certain features that are already present in the Linux Kernel
• Namespaces → isolate what processes can see
• Cgroups → control CPU, memory and resource usage
• Capabilities → fine-grained permissions
• Seccomp → filter system calls
• Mount / OverlayFS → filesystem isolation
• LSMs → security enforcement modules

Namespaces: Isolation of What a Process Can See
▷ Each process gets its own isolated view of the system
↪ Same kernel, but different resource views
• PID Namespace → each container has its own process tree (PID 1)
• NET Namespace → each has separate network interfaces, IP addresses, ports
• MNT Namespace → separate FS mount points, allowing container to have its own root FS
• UTS Namespace → separate hostname and domain name for each container
• IPC Namespace → each container has separate inter-process communication resources
PID → Process ID isolation
NET → Network stack
MNT → Mount points
UTS → UNIX Timesharing System
IPC → Inter-Process Communication
Common Linux Namespaces
PID
Isolates process ID space, making processes inside the container appear as PID 1.
  • Independent process tree
  • PID 1 inside container
NET
Provides independent network stack with interfaces, routing tables, and firewall rules.
  • Isolated network interfaces
  • Independent IP addresses
MNT
Isolates mount points, allowing containers to have their own filesystem view.
  • Independent mount table
  • Filesystem isolation
UTS
Allows systems to have different host and domain names inside containers.
  • Hostname isolation
  • Domain name separation
IPC
Isolates System V IPC objects and POSIX message queues between containers.
  • Shared memory isolation
  • Message queue separation

runc: The Executor
▷ runc is a low-level runtime designed to run containers using Linux primitives
↪ It is the component that actually runs containers
• Uses Namespaces → isolates processes and resources
• Uses cgroups → controls CPU, memory, I/O usage
• Creates container from OCI bundle = root filesystem + config.json
• Directly interacts with Linux kernel
• Extremely lightweight → just a thin wrapper over kernel features
Namespaces → Isolation

Kernel → System calls
Minimal → Fast execution

Docker Image: Layers
• Docker images are built in layers
• Each instruction creates a new layer
• Layers are read-only
• Containers add a writable temporary layer on top of it
   w/o modifying original slide
• Uses copy-on-write mechanism for efficient storage
   & fast builds
// DOCKERFILE
$ FROM ubuntu
// Creates a layer for base ubuntu
$ RUN apt update
// Creates layer for updated packages
$ RUN apt install python3
// Layer for installed python
$ COPY app.py /app/
// Layer for app code
Docker Networking Internals
• Each container gets its own network namespace
• By Default, containers are connected via a bridge network
• Each container gets its own IP address
• Uses veth pairs to connect container ↔ host
• iptables are used for NAT, port forwarding, and filtering

“    But if all the containers are using the same kernel, Docker's probably unsafe right?
Docker Security: Isolation on Shared Kernel
• Containers share the same Linux kernel
↪ But remain isolated using kernel features
• Namespaces → isolate processes, network, filesystem
• cgroups → limit CPU, memory, I/O usage
• Capabilities → restrict root privileges
• Seccomp → filters system calls
• AppArmor / SELinux → enforce security policies
Namespaces → Isolation
cgroups → Resource control
Capabilities → Limited root access
Seccomp → System call filtering
LSMs → Mandatory access control
→ Strong isolation on shared kernel
